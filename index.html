Build-Bench Challenge

Cross-Architecture Package Build Repair

Can language-model systems diagnose and repair real software packages that build on one instruction set architecture but fail on another? Build-Bench evaluates the answer with clean patch application and executable builds, not reference-patch similarity.

Competition task

Repair the package, then prove it builds

A task instance starts from a package that succeeds on a source architecture and fails on the target architecture. A participant system consumes the failed case and produces a repair patch. The evaluator applies that patch to a clean copy and rebuilds the package on the target architecture.

01

Inspect

Read the manifest, package layout, build scripts, metadata, and failed log.

02

Diagnose

Localize architecture-specific assumptions, dependency faults, or packaging errors.

03

Repair

Produce a focused unified diff that modifies only permitted package files.

04

Verify

Apply the patch in a clean workspace and run the target-architecture build validator.

Case input

What each released case contains

  • Package specification files such as RPM .spec files
  • Source archives, existing patches, service files, and package metadata
  • The original failed build log
  • Source and target architecture labels
  • Checksums, constraints, and optional immutable metadata
Repair scope

What systems may need to change

  • Architecture guards and dependency declarations
  • Compiler flags, build scripts, and generated configuration
  • Source-level architecture assumptions
  • Target-sensitive tests and runtime expectations
  • Installed files, packaging macros, and scriptlets
Dataset

A real, system-level repair corpus

The competition will be derived from the Build-Bench research corpus collected from public openSUSE build scenarios. Every retained package was rebuilt to confirm that the cross-architecture failure was reproducible.

The latest paper reports 163 x86_64-to-aarch64 cases and 105 reverse cases. Final competition split counts will be frozen before launch.

Evaluation

Executable outcomes are the oracle

A repair counts only when the patch applies cleanly, respects policy, and the package completes the official build validator on the target architecture.

Primary metric
SuccessRate = successfully built casesevaluated cases

Textual similarity to an organizer patch is not used as the success criterion. The package must build in the configured environment.

Tie-breakers, in order
  1. 01Fewer invalid patches
  2. 02Smaller total patch size
  3. 03Fewer modified files
  4. 04Lower organizer-side build time

Verify input

Apply patch

Use standard patch tools and record whether application succeeds.

Check policy

Reject forbidden paths, evaluator metadata changes, and invalid repairs.

Public validation feedback

Participants will receive per-case outcomes and logs on public splits. The proposal targets completion within a few hours using a reduced public scope and conservative per-case timeouts.

Hidden final evaluation

Organizers will use a frozen evaluator, the same timeout policy, prebuilt images, and cached dependencies where possible. Only aggregate results and a small diagnostic set will be released after the deadline.

Participation

Two submission modes, one build criterion

Patch output is the proposed main ranking mode. A containerized-agent mode is optional and will be activated only after the patch infrastructure and sandbox are stable.

Rules and integrity

Repair the build, do not bypass it

The primary leaderboard is intended for LLM-based and agentic systems. Final rules will be versioned before submissions open, but the accepted proposal already defines the following integrity requirements.

Disclose the system

Report the model and major tool sources used to generate repairs. Purely non-LLM systems may appear as labeled comparisons or baselines.

Do not exploit the evaluator

No hard-coded hidden solutions, evaluator-bug exploitation, validator disabling, essential-test removal, or dummy package outputs.

Modify only allowed paths

Inputs and evaluator metadata are immutable. The final evaluator will audit modified paths and reject forbidden changes.

Respect hidden-evaluation isolation

Network access is disabled unless an optional container mode explicitly supports declared model APIs under a published policy.

Expect audit and reruns

Organizers may manually inspect successful, unusually small, or suspicious repairs and rerun affected submissions after evaluator fixes.

FAQ

What participants need to know

Confirmed details will be versioned here as the starter kit and submission platform are released. Items marked as planned are not yet active services.

What is the primary competition artifact?

The proposed main mode accepts one unified diff per case. Teams generate patches with their own LLM or agentic workflow, then organizers apply and evaluate them in a controlled environment.

Must we submit a complete agent?

No. Containerized agent submission is optional and planned only after the patch-output evaluator and sandbox are stable. It will not replace the main patch ranking.

Can we use closed models and external APIs?

Patch-generation workflows may use them when models, APIs, retrieval sources, and major tools are disclosed. Network policy for organizer-run containers will be published separately.

How is a successful repair determined?

The patch must apply cleanly, avoid forbidden evaluator files, pass policy checks, and complete the target-architecture build validator without bypassing required build or test steps.

Will the hidden test set be released?

No. The hidden set remains organizer-controlled through final evaluation. Only aggregate results and a limited diagnostic breakdown will be released after the deadline.

How often may a team submit?

The final limit is to be announced. The proposal allows rate limits to control build resources and reduce overfitting to the public validation set.

Where will submissions run?

The preferred public platform is a hosted leaderboard such as EvalAI. If long-running builds make that impractical, submissions may use GitHub releases with organizer-server validation on a fixed cadence. The platform decision is not yet final.

What baselines will be provided?

The proposal lists four: no-op, spec-only heuristic, single-shot LLM, and an iterative MCP-based Build-Bench agent with structure extraction, file inspection, patch generation, and build verification.
